Tenchi

Tenchi is a small, contract-first Python framework for typed JSON APIs. Contracts define the HTTP boundary, plain async functions implement use cases, and frozen dataclasses carry explicitly wired dependencies. Tenchi requires Python 3.12 or newer and is currently pre-1.0.

This guide follows one todo API from its contract through application wiring, then adds errors, clients, alternate entrypoints, testing, and operational tooling.

Start a new application:

uvx tenchi new my_app
cd my_app
uv sync
uv run tenchi dev

To add Tenchi to an existing project instead, run uv add tenchi.

Contracts

A contract is the source of truth for an endpoint’s method, path, inputs, response body, successful response headers, status, and application errors. Pydantic validates these annotations at the HTTP boundary. 

from pydantic import BaseModel, Field
from tenchi.contracts import contract

class CreateTodo(BaseModel):
    title: str = Field(min_length=1)

class Todo(BaseModel):
    id: str
    title: str
    completed: bool

class CreatedTodoHeaders(BaseModel):
    location: str = Field(alias="Location")

create_todo_contract = contract(
    method="POST",
    path="/todos",
    request=CreateTodo,
    response=Todo,
    response_headers=CreatedTodoHeaders,
    status=201,
)

Use cases and ports

The application owns its ports as protocols. A frozen context carries their implementations, and use cases remain ordinary async functions with no dependency on the HTTP runtime. 

from dataclasses import dataclass
from typing import Protocol

class TodoRepository(Protocol):
    async def create(self, *, title: str) -> Todo: ...
    async def get(self, todo_id: str) -> Todo | None: ...
    async def list(self) -> list[Todo]: ...

@dataclass(frozen=True, slots=True)
class AppContext:
    todos: TodoRepository

async def create_todo(request: CreateTodo, context: AppContext) -> Todo:
    return await context.todos.create(title=request.title)

Application wiring

Routes bind contracts to use cases. The application root composes those routes and supplies concrete port implementations; here, repository implements TodoRepository. 

from tenchi.health import health_route
from tenchi.openapi import openapi_route
from tenchi.routes import route, route_group
from tenchi.server import create_app

def create_todo_headers(todo: Todo) -> CreatedTodoHeaders:
    return CreatedTodoHeaders(Location=f"/todos/{todo.id}")

todo_routes = route_group(
    route(
        create_todo_contract,
        create_todo,
        response_headers=create_todo_headers,
    ),
)

routes = route_group(
    todo_routes,
    openapi_route(todo_routes, title="Todos", version="1.0.0"),
    health_route(),
)

app = create_app(
    routes=routes,
    context_factory=lambda: AppContext(todos=repository),
)

The synchronous projector keeps HTTP metadata at the route boundary; Tenchi validates its fixed, scalar fields before the request scope commits. Aliases are the header names on the wire. create_app() also accepts lifespan resources, hooks, middleware, and request-body limits.

Contracts are private by default. Set public=True on an operation that authentication hooks should exempt, then inspect info.contract.public in the hook. OpenAPI uses the same metadata to remove global security from that operation. health_route() and openapi_route() are public by default and accept public=False when they should be protected. The metadata does not authenticate requests by itself. Authentication belongs in boundary hooks; authorization stays in use cases and pure policy functions.

Errors

Define stable application errors and declare them on a contract or route group. Undeclared application errors become framework-owned 500s. 

from tenchi.errors import AppError, ErrorDef

class GetTodoParams(BaseModel):
    todo_id: str

todo_not_found = ErrorDef(
    code="TODO_NOT_FOUND",
    status=404,
    message="Todo not found",
)

get_todo_contract = contract(
    method="GET",
    path="/todos/{todo_id}",
    params=GetTodoParams,
    response=Todo,
    errors=(todo_not_found,),
)

async def get_todo(params: GetTodoParams, context: AppContext) -> Todo:
    todo = await context.todos.get(params.todo_id)
    if todo is None:
        raise AppError(todo_not_found, details={"todo_id": params.todo_id})
    return todo

The same honesty rule applies to boundary hooks: an authentication failure maps to its application status only when the matched contract or route group declares that error.

Client and OpenAPI

The async client uses the same contract to serialize inputs and validate responses, so the server and client do not need separate schema declarations. 

from tenchi.client import Client

async with Client(base_url="https://api.example.com") as client:
    created = await client.call_with_response(
        create_todo_contract,
        request=CreateTodo(title="Buy milk"),
    )
    todo = created.body
    location = created.headers.location
    status = created.http_response.status_code

call() still returns only the validated body. Both client methods validate declared success headers and the response Content-Type. The server likewise requires request bodies to match their contract media type, returning a framework-owned 415 before decoding a missing or mismatched type. Charset-qualified text contracts are encoded and decoded strictly in both directions, and unsupported declared charsets fail when the contract is built.

openapi_route() serves an OpenAPI 3.1 document from the same declarations, including successful headers and framework-owned error responses.

Multiple responses and runtime controls

Routes that can succeed in more than one way declare response definitions. Their body and header types determine the aggregate client result, while a synchronous presenter selects the status-specific response and the use case keeps returning domain data. Here, put_todo returns a PutTodoResult. 

from dataclasses import dataclass
from tenchi.responses import PresentedResponse, present, response

@dataclass(frozen=True, slots=True)
class PutTodoResult:
    todo: Todo
    created: bool

created_response = response(Todo, status=201)
existing_response = response(Todo, status=200)

put_todo_contract = contract(
    method="PUT",
    path="/todos",
    request=CreateTodo,
    responses=(created_response, existing_response),
    timeout=5.0,
)

def present_put(result: PutTodoResult) -> PresentedResponse:
    definition = created_response if result.created else existing_response
    return present(definition, result.todo)

route(put_todo_contract, put_todo, present=present_put)

When one status permits alternative top-level body schemas, pass them separately so Pyright preserves the exact union: response(Todo, str, status=200). Nested unions retain their ordinary spelling, such as response(list[Todo | str], status=200). Each definition has one fixed object-shaped header schema; differing header shapes belong in separate definitions.

A response defined with passthrough=True may instead carry a Starlette streaming, file, or redirect response. Tenchi preserves the response while checking its contract-owned status, media-type parameters, and declared headers. Definitions without a body accept only a concrete, provably empty response; streaming definitions declare their body type. The typed client validates the selected definition and exposes it as ClientResponse.definition.

A contract’s timeout= cooperatively cancels overdue work, waits for request-scope cleanup, and returns a framework 504 even if application code catches the cancellation. At the application boundary, create_app(observers=...) delivers immutable request outcomes for metrics or tracing after matched requests finalize; observer failures are logged and cannot alter the response.

Pagination

Subclass PageQuery for filters, return Page[Todo], and build the result with page(). 

from tenchi.pagination import Page, PageQuery, page

class ListTodosQuery(PageQuery):
    completed: bool | None = None

async def list_todos(
    query: ListTodosQuery,
    context: AppContext,
) -> Page[Todo]:
    items = await context.todos.list()
    start = query.offset
    window = items[start : start + query.limit]
    return page(window, total=len(items), query=query)

Workers and scripts

HTTP is only one caller of a use case. execute() validates job data against the same request annotation and applies the server’s context-scoping rules without introducing another contract. 

from tenchi.execution import execute

async def handle_job(payload: bytes, context: AppContext) -> Todo:
    return await execute(
        create_todo,
        request_json=payload,
        context=context,
    )

Testing

Test use cases directly with memory implementations of their ports. For boundary behavior, open_client() provides the typed client in-process and open_http() exposes raw httpx responses. Both run the application lifespan. 

from tenchi.testing import open_client, open_http

async with open_client(app) as client:
    todo = await client.call(
        create_todo_contract,
        request=CreateTodo(title="Buy milk"),
    )

async with open_http(app) as http:
    response = await http.get("/health")
    assert response.status_code == 200

CLI

The CLI scaffolds structural conventions, inspects the composed application, and keeps its OpenAPI contract reviewable in version control. 

uvx tenchi new my_app
cd my_app
uv run tenchi make feature notes
uv run tenchi make use-case notes create_note
uv run tenchi routes
uv run tenchi openapi
uv run tenchi openapi --diff openapi.json
uv run tenchi openapi --check openapi.json
uv run tenchi openapi --write openapi.json
uv run tenchi doctor
uv run tenchi dev

openapi --write stores canonical, key-sorted JSON. Before accepting a changed snapshot, use openapi --diff to classify breaking, additive, metadata-only, and unknown changes. Breaking and unknown changes fail; additive and metadata-only changes pass. Add --diff-format json for machine-readable output, and use openapi --check for exact drift detection in tests or CI. Use the same routes, metadata, and security options in every command.

Run the diff before replacing its baseline. CI should read that baseline from the merge base or previous release, not from the same change being checked. For programmatic checks, import analyze_openapi_compatibility from tenchi.compatibility.
